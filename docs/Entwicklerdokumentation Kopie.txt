        <h2>Einstiegspunkt & App-Aufbau</h2>
      </div>

      <h3>main.tsx</h3>
      <p><code>main.tsx</code> ist der Einstiegspunkt. Er rendert <code>App</code> in <code>#root</code> und initialisiert den Service Worker:</p>
      <pre><code>import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import { initServiceWorker } from './utils/serviceWorker';
import App from './App';
import './index.css';

initServiceWorker();
createRoot(document.getElementById('root')!).render(
  &lt;StrictMode&gt;&lt;App /&gt;&lt;/StrictMode&gt;
);</code></pre>

      <h3>App.tsx – Provider-Hierarchie und Routing</h3>
      <p><code>App.tsx</code> legt die vollständige Komponentenhierarchie fest. Alle Seiten sind durch <code>LearningProvider</code> umschlossen, der den globalen State und die Geschäftslogik bereitstellt:</p>

      <div class="flow">
        <div class="flow-row">
          <div class="flow-box accent">LearningProvider</div>
          <div class="flow-arrow">→</div>
          <div class="flow-box">BrowserRouter</div>
          <div class="flow-arrow">→</div>
          <div class="flow-box">Layout (Bottom-Nav)</div>
          <div class="flow-arrow">→</div>
          <div class="flow-box purple">Seite (Route)</div>
        </div>
      </div>

      <p>Alle fünf Seiten werden unter dem gemeinsamen <code>Layout</code> gerendert. Unbekannte Routen werden per <code>Navigate</code> auf <code>/</code> umgeleitet.</p>

      <h3>index.css</h3>
      <p>Enthält ausschließlich die CSS-Flip-Animation der Karteikarte (<code>.fc-scene</code>, <code>.fc-inner</code>, <code>.fc-face</code>, <code>.fc-back</code>) sowie Reset-Regeln für <code>html/body</code> und die iOS-Safe-Area-Variable. Alle anderen Styles sind als Inline-Styles in den jeweiligen Komponenten definiert.</p>
    </section>

      <p>React Router v6 mit <code>BrowserRouter</code>. Alle Routen sind flach unter einem einzigen <code>Layout</code>-Element verschachtelt.</p>

      <div class="table-wrap">

      <p>Die <code>Layout</code>-Komponente rendert die Bottom-Navigation mit fünf <code>NavLink</code>-Elementen und zeigt den aktiven Tab durch eine gelbe Farbe hervor. Der eigentliche Seiteninhalt wird via <code>&lt;Outlet /&gt;</code> eingesetzt.</p>
    </section>

    <!-- STATE MANAGEMENT -->
    <section id="state">
      <div class="section-header">
        <div class="section-num">5</div>
        <h2>State Management</h2>
      </div>
      <p>Der gesamte Anwendungsstatus lebt in einem einzigen React Context: <code>LearningContext.tsx</code>. Es gibt keinen externen State-Manager (kein Redux, kein Zustand).       <h3>Persistenz des State</h3>
      <p>Der gesamte State wird als ein einziges JSON-Objekt unter dem Schlüssel <code>vokabeltrainer_state</code> gespeichert. Schreibvorgänge werden mit einem <strong>300 ms Debounce</strong> ausgeführt, um bei raschen State-Änderungen keine unnötigen IndexedDB-Zugriffe zu erzeugen. Beim Start liest ein <code>useEffect</code> den gespeicherten Zustand und hydriert den State; bis dieser Ladevorgang abgeschlossen ist, rendert der Provider <code>null</code> (kein Flackern).</p>

      <h3>Datenmigration</h3>
      <p>Beim ersten Start migriert der Context ältere Datenstände automatisch:</p>
      <ul>
        <li>localStorage → IndexedDB (<code>migrateFromLocalStorage</code>)</li>
        <li>Altes <code>customVocabulary</code>-Format (einzelne Liste) → <code>customVocabularyEN</code> + <code>customVocabularyES</code></li>
        <li>Altes <code>trainingLog</code>-Format (flaches Array) → <code>Record&lt;Language, string[]&gt;</code></li>
        <li>Altes <code>dailyStats</code>-Format (sprachunabhängig) → per-Language</li>
      </ul>

      <div class="section-header">
        <div class="section-num">6</div>
        <h2>Datenschicht</h2>
      </div>

      <h3>VocabularyItem (vocabulary_types.ts)</h3>
      <p>Das zentrale Datenmodell für eine Vokabel:</p>
      <pre><code>interface VocabularyItem {
  id:           string;    // Eindeutige ID; eingebaut: sprechend, custom: "custom_{timestamp}"
  german:       string;    // Das deutsche Wort
  translation:  string;    // Übersetzung in der Zielsprache
  emoji:        string;    // Emoji zur visuellen Veranschaulichung
  category:     Category;  // Thematische Kategorie (z.B. "Tiere")
  inflections?: string;    // Beugungsformen, optional (z.B. "[dogs]" oder "[lief, gelaufen]")
  isCustom?:    boolean;   // true = eigene Vokabel, false/undefined = eingebaut
}</code></pre>

      <h3>CardProgress (LearningContext.tsx)</h3>
      <p>Fortschrittsstand einer einzelnen Karte für eine bestimmte Sprache:</p>
      <pre><code>interface CardProgress {
  box:           number;          // Fach 1–6
  lastReviewed:  string | null;   // ISO-Zeitstempel der letzten Bewertung; null = noch nie gesehen
  nextDate:      string | null;   // YYYY-MM-DD des nächsten fälligen Tages; null = Fach 6 (fertig)
  correctCount:  number;          // Gesamtzahl richtiger Antworten
  incorrectCount: number;         // Gesamtzahl falscher Antworten
}</code></pre>

      <p>Der Fortschritts-Schlüssel im <code>progress</code>-Objekt lautet immer <code>{vocabId}_{lang}</code>, z.B. <code>apple_english</code>. Dadurch ist der Fortschritt für jede Sprache separat gespeichert – eine englische Vokabel und eine spanische mit gleicher ID stören sich nicht.</p>

      <h3>Vokabeldaten (vocabulary_en.ts / vocabulary_es.ts)</h3>
      <p>Die eingebauten Vokabeln sind statische TypeScript-Arrays (<code>VOCABULARY_EN</code>, <code>VOCABULARY_ES</code>), die beim Build direkt in das Bundle kompiliert werden. Um neue eingebaute Vokabeln hinzuzufügen, wird der entsprechende Array in der jeweiligen Datei erweitert.</p>

      <h3>Kategorien</h3>
      <p><code>ALL_CATEGORIES</code> in <code>vocabulary.ts</code> ist das einzige definierte Array der 13 Kategorien. Dieses Array wird sowohl für Filter-UI als auch für das Formular beim Hinzufügen eigener Vokabeln verwendet. Neue Kategorien werden hier ergänzt und müssen auch in den <code>Category</code>-Typ in <code>vocabulary_types.ts</code> aufgenommen werden.</p>
    </section>

    <section id="storage">
      <div class="section-header">
        <div class="section-num">7</div>
        <h2>Persistenz (storage.ts)</h2>
      </div>
      <p><code>storage.ts</code> kapselt den gesamten Zugriff auf den persistenten Speicher hinter drei einfachen async-Funktionen:</p>

      <div class="table-wrap">
        <table>
          <thead><tr><th>Funktion</th><th>Beschreibung</th></tr></thead>
          <tbody>
            <tr><td><code>storageGet(key)</code></td><td>Liest einen String aus IndexedDB. Fallback auf localStorage bei Fehler.</td></tr>
            <tr><td><code>storageSet(key, value)</code></td><td>Schreibt einen String nach IndexedDB. Fallback auf localStorage.</td></tr>
            <tr><td><code>migrateFromLocalStorage(key)</code></td><td>Einmalige Migration: liest Key aus localStorage, schreibt nach IndexedDB, entfernt aus localStorage.</td></tr>
          </tbody>
        </table>
      </div>

      <p><strong>Warum IndexedDB statt localStorage?</strong> Safari auf iOS löscht den localStorage aggressiv bei Speicherdruck oder wenn die App längere Zeit nicht genutzt wurde. IndexedDB ist stabiler und für PWAs die empfohlene Speicherstrategie. Der localStorage-Fallback bleibt als Sicherheitsnetz für Browser ohne IndexedDB-Unterstützung.</p>

      <div class="callout warn">
        <strong>Datenstruktur beachten</strong>
        Der gesamte State wird als ein einziger JSON-String gespeichert. Bei strukturellen Änderungen am State-Interface (z.B. neue Felder) muss die Migrations-Logik in <code>LearningContext.tsx</code> angepasst werden, damit ältere gespeicherte Zustände korrekt gelesen werden können.
      </div>
    </section>

    <!-- SPACED REPETITION -->
    <section id="spaced">
      <div class="section-header">
        <div class="section-num">8</div>
        <h2>Spaced-Repetition-Logik</h2>
      </div>
      <p>Die Spaced-Repetition-Logik ist vollständig in <code>LearningContext.tsx</code> implementiert. Die Intervalle sind als Konstante in <code>constants/theme.ts</code> definiert:</p>
      <pre><code>export const BOX_INTERVALS = [0, 1, 3, 7, 14, Infinity];
// Index 0 = Fach 1, Index 5 = Fach 6</code></pre>

      <h3>isCardDue(progress)</h3>
      <p>Bestimmt, ob eine Karte heute fällig ist:</p>
      <ul>
        <li>Fach 6 → nie fällig (immer <code>false</code>)</li>
        <li><code>lastReviewed === null</code> → neue, noch nie gesehene Karte → <strong>nicht</strong> fällig über <code>getDueCards</code>, wird stattdessen über <code>getNewCards</code> eingeführt</li>
        <li><code>nextDate</code> vorhanden → Vergleich mit heutigem Datum (lokale Zeitzone, Format <code>YYYY-MM-DD</code>)</li>
        <li>Legacy-Karten ohne <code>nextDate</code> → Rückfall auf intervalbasierte Berechnung</li>
      </ul>

      <h3>markCard(id, lang, correct)</h3>
      <p>Kernfunktion für die Bewertung einer Karte:</p>
      <ul>
        <li><strong>Richtig:</strong> <code>newBox = min(box + 1, 6)</code>. <code>nextDate</code> = heute + Intervall des neuen Fachs.</li>
        <li><strong>Falsch:</strong> <code>newBox = 1</code>. <code>nextDate</code> = morgen (Fach-1-Karten sollen nicht sofort wieder erscheinen).</li>
        <li>Fach 6 bei richtiger Antwort: <code>nextDate = null</code> (nie wieder fällig).</li>
        <li>Aktualisiert <code>dailyStats</code> (Zähler für fällige Karten heute) und <code>dailyNewStats</code> (neue Karten heute).</li>
      </ul>

      <h3>getDueCards(lang)</h3>
      <p>Gibt fällige Karten zurück, sortiert nach: Box aufsteigend, dann <code>nextDate</code> aufsteigend. Das <code>dailyCardLimit</code> wird gegen <code>dailyStats</code> geprüft – bereits bewertete Karten des Tages reduzieren das verbleibende Limit.</p>

      <h3>getNewCards(lang)</h3>
      <p>Gibt Karten zurück, bei denen <code>lastReviewed === null</code> ist (noch nie gesehen). Das <code>dailyNewCardLimit</code> wird gegen <code>dailyNewStats</code> geprüft.</p>
    </section>

      <h3>Workbox-Strategie (vite.config.ts)</h3>
      <div class="table-wrap">
        <table>
          <thead><tr><th>Asset-Typ</th><th>Strategie</th><th>Cache-Dauer</th></tr></thead>
          <tbody>
            <tr><td>JS, CSS, Fonts, Bilder (Precache)</td><td>CacheFirst</td><td>1 Jahr</td></tr>

      <p><code>clientsClaim: true</code> und <code>skipWaiting: true</code> stellen sicher, dass ein neuer Service Worker sofort alle offenen Tabs übernimmt. Das ist auf iOS besonders wichtig, wo Tabs den alten Worker sonst sehr lange behalten können.</p>

      <h3>serviceWorker.ts – iOS-Besonderheiten</h3>
      <p>Safari beendet den Service Worker nach ca. 30 Sekunden im Hintergrund. <code>serviceWorker.ts</code> behandelt drei Szenarien:</p>
      <ul>
        <li><strong>Stündlicher Update-Check:</strong> Nur wenn online (<code>navigator.onLine</code>), um Fehler beim Offline-Betrieb zu vermeiden.</li>
        <li><strong><code>visibilitychange</code>-Listener:</strong> Wenn die App aus dem Hintergrund zurückkommt und online ist, wird ein Update-Check ausgelöst. Falls kein SW mehr registriert ist (iOS hat ihn beendet), wird einmalig ein Reload versucht.</li>
        <li><strong><code>pageshow</code>-Listener:</strong> Behandelt iOS bfcache (Back-/Forward-Cache nach App-Switcher) und löst ebenfalls einen Update-Check aus.</li>
      </ul>

      <h3>swOverlay.ts</h3>
      <p>Hilfsfunktion für die Diagnose. Blendet ein temporäres, terminal-ähnliches Overlay ein (grüner Text auf schwarzem Hintergrund), das den aktuellen SW-Zustand, Online-Status und Cache-Belegung anzeigt. Verschwindet nach 8 Sekunden oder bei Antippen. Nur über das SWStatusPanel in den Einstellungen erreichbar.</p>

      <h3>Neue Vokabeln zu einer bestehenden Sprache hinzufügen</h3>
      <p>Die Datei <code>src/data/vocabulary_en.ts</code> (bzw. <code>vocabulary_es.ts</code>) erweitern:</p>

{
  id: 'eindeutige_id',      // Muss eindeutig sein – z.B. 'tree', 'to_swim_en'
  german: 'der Baum',
  translation: 'tree',
  emoji: '🌳',

      <p>Die App unterstützt beliebig viele Sprachen. Eine Schritt-für-Schritt-Anleitung dazu ist in <a href="neue-sprache-hinzufuegen.md" style="color: var(--accent2); font-weight: 600;">docs/neue-sprache-hinzufuegen.md</a> beschrieben.</p>
      <p>Zusammengefasst sind folgende Schritte nötig:</p>
      <ol>
        <li><code>src/data/vocabulary_XX.ts</code> anlegen und mit <code>VOCABULARY_XX</code> befüllen.</li>
        <li>Re-Export in <code>src/data/vocabulary.ts</code> ergänzen.</li>
        <li>Neuen Spracheintrag in <code>src/constants/languages.ts</code> zu <code>LANGUAGE_CONFIG</code> hinzufügen (Label, Flag, Abbr, speechCode, placeholder).</li>
        <li><code>Language</code>-Typ in <code>languages.ts</code> um den neuen Wert erweitern.</li>
        <li>Im <code>LearningContext</code>: Initialen State um die neue Sprache ergänzen (<code>customVocabularyXX</code>, <code>dailyStats</code>, <code>dailyNewStats</code>, <code>trainingLog</code>).</li>
      </ol>

      <div class="callout info">
        <strong>Kein zentrales Routing-Update nötig</strong>
        Seiten wie Home, Statistics und Vocabulary lesen Sprachen dynamisch aus <code>ALL_LANGUAGES</code> (aus <code>constants/languages.ts</code>). Nach dem Hinzufügen einer Sprache erscheint sie automatisch in der Sprachauswahl und in den Filtern, ohne dass Seiten-Code angepasst werden muss.
      </div>
