Einstiegspunkt & App-Aufbau

Startsequenz beim App-Laden

Wenn der Browser index.html lädt, läuft folgende Kette ab:

main.tsx wird ausgeführt
Vite injiziert den Bundle-Einstiegspunkt. initServiceWorker() wird synchron aufgerufen, bevor React mountet – damit der Service Worker so früh wie möglich registriert ist.
LearningProvider mountet
Der Context-Provider setzt den initialen State (leere Felder, Standardwerte). Solange der gespeicherte Zustand noch nicht geladen ist, gilt loaded = false und der Provider rendert null – keine UI erscheint, kein Flackern.
Async: IndexedDB-Migration & Laden
Ein useEffect im Provider führt migrateFromLocalStorage() aus (einmalige Datenmigration), liest dann den Rohwert aus IndexedDB via storageGet() und parst das JSON. Falls alte Felder fehlen oder das Format veraltet ist, werden Migrations-Schritte nachgezogen (z.B. flaches dailyStats → per-Language, altes customVocabulary → Split EN/ES).
State-Hydration: setState(parsed)
Der vollständige gespeicherte Zustand ersetzt den Initial-State. Danach wird setLoaded(true) gesetzt.
UI rendert erstmals sichtbar
Sobald loaded = true, gibt der Provider seine Children frei. React Router matcht die aktuelle URL und rendert die passende Seite. Ab jetzt beobachtet ein zweiter useEffect State-Änderungen und schreibt sie mit 300 ms Debounce zurück in IndexedDB.

State-Update-Zyklus

Jede Benutzeraktion durchläuft immer denselben Weg – von der Komponente bis zur Festplatte:

1
Komponente ruft Context-Methode auf
z.B. markCard('apple', 'english', true) in Learn.tsx
2
Methode ruft setState(s => …) auf
Alle Methoden im Context verwenden den Functional-Update-Style um Race Conditions bei schnellen Klicks zu verhindern. Der State wird nie direkt mutiert – immer ein neues Objekt mit Spread-Syntax.
3
React re-rendert betroffene Komponenten
Alle Seiten/Komponenten, die useLearning() verwenden, erhalten den aktualisierten Context und rendern neu.
4
Save-Effect startet 300 ms Debounce
Ein useEffect([state, loaded]) reagiert auf jede State-Änderung. Er startet (oder verschiebt) einen Timeout. Erst wenn 300 ms lang keine weitere Änderung kommt, wird geschrieben.
5
storageSet(key, JSON.stringify(state))
Der vollständige State wird als ein einziger JSON-String in IndexedDB geschrieben. Bei Fehler greift der Fallback auf localStorage.

Persistenz des State

Der gesamte State wird als ein einziges JSON-Objekt unter dem Schlüssel vokabeltrainer_state gespeichert. Beim Start liest ein useEffect den gespeicherten Zustand und hydriert den State; bis dieser Ladevorgang abgeschlossen ist, rendert der Provider null (kein Flackern).

Datenmigration

Beim ersten Start migriert der Context ältere Datenstände automatisch:

  • localStorage → IndexedDB (migrateFromLocalStorage)
  • Altes customVocabulary-Format (einzelne Liste) → customVocabularyEN + customVocabularyES
  • Altes trainingLog-Format (flaches Array) → Record<Language, string[]>
  • Altes dailyStats-Format (sprachunabhängig) → per-Language

Datenschicht

VocabularyItem (vocabulary_types.ts)

Das zentrale Datenmodell für eine Vokabel:

interface VocabularyItem {
  id:           string;    // Eindeutige ID; eingebaut: sprechend, custom: "custom_{timestamp}"
  german:       string;    // Das deutsche Wort
  translation:  string;    // Übersetzung in der Zielsprache
  emoji:        string;    // Emoji zur visuellen Veranschaulichung
  category:     Category;  // Thematische Kategorie (z.B. "Tiere")
  inflections?: string;    // Beugungsformen, optional (z.B. "[dogs]" oder "[lief, gelaufen]")
  isCustom?:    boolean;   // true = eigene Vokabel, false/undefined = eingebaut
}

CardProgress (LearningContext.tsx)

Fortschrittsstand einer einzelnen Karte für eine bestimmte Sprache:

interface CardProgress {
  box:           number;          // Fach 1–6
  lastReviewed:  string | null;   // ISO-Zeitstempel der letzten Bewertung; null = noch nie gesehen
  nextDate:      string | null;   // YYYY-MM-DD des nächsten fälligen Tages; null = Fach 6 (fertig)
  correctCount:  number;          // Gesamtzahl richtiger Antworten
  incorrectCount: number;         // Gesamtzahl falscher Antworten
}

Der Fortschritts-Schlüssel im progress-Objekt lautet immer {vocabId}_{lang}, z.B. apple_english. Dadurch ist der Fortschritt für jede Sprache separat gespeichert – eine englische Vokabel und eine spanische mit gleicher ID stören sich nicht.

Vokabeldaten (vocabulary_en.ts / vocabulary_es.ts)

Die eingebauten Vokabeln sind statische TypeScript-Arrays (VOCABULARY_EN, VOCABULARY_ES), die beim Build direkt in das Bundle kompiliert werden. Um neue eingebaute Vokabeln hinzuzufügen, wird der entsprechende Array in der jeweiligen Datei erweitert.

Kategorien

ALL_CATEGORIES in vocabulary.ts ist das einzige definierte Array der 13 Kategorien. Dieses Array wird sowohl für Filter-UI als auch für das Formular beim Hinzufügen eigener Vokabeln verwendet. Neue Kategorien werden hier ergänzt und müssen auch in den Category-Typ in vocabulary_types.ts aufgenommen werden.

Persistenz (storage.ts)

storage.ts kapselt den gesamten Zugriff auf den persistenten Speicher hinter drei einfachen async-Funktionen:

FunktionBeschreibung
storageGet(key)Liest einen String aus IndexedDB. Fallback auf localStorage bei Fehler.
storageSet(key, value)Schreibt einen String nach IndexedDB. Fallback auf localStorage.
migrateFromLocalStorage(key)Einmalige Migration: liest Key aus localStorage, schreibt nach IndexedDB, entfernt aus localStorage.

Lese-Ablauf: storageGet()

1
IndexedDB öffnen – Datenbank VokabelTrainerDB, Object Store keyval, Version 1. Wird beim ersten Aufruf automatisch angelegt.
2
IDBRequest: store.get(key) – Asynchroner Lesevorgang. Bei Erfolg wird der gespeicherte String zurückgegeben (oder null falls Key unbekannt).
!
Bei Fehler: IndexedDB nicht verfügbar (privater Modus, alter Browser) → Fallback: localStorage.getItem(key) wird direkt zurückgegeben.

Schreib-Ablauf: storageSet()

1
IndexedDB öffnen – gleicher Datenbankname, Transaktion im 'readwrite'-Modus.
2
store.put(value, key) – Schreibt den Wert. Existiert der Key bereits, wird er überschrieben (upsert-Semantik).
!
Bei Fehler: Fallback: localStorage.setItem(key, value). localStorage hat ein 5 MB Limit – bei sehr großen Vokabellisten kann dies zum Problem werden.

Warum IndexedDB statt localStorage? Safari auf iOS löscht den localStorage aggressiv bei Speicherdruck oder wenn die App längere Zeit nicht genutzt wurde. IndexedDB ist stabiler und für PWAs die empfohlene Speicherstrategie. Der localStorage-Fallback bleibt als Sicherheitsnetz für Browser ohne IndexedDB-Unterstützung.

Datenstruktur beachten Der gesamte State wird als ein einziger JSON-String gespeichert. Bei strukturellen Änderungen am State-Interface (z.B. neue Felder) muss die Migrations-Logik in LearningContext.tsx angepasst werden, damit ältere gespeicherte Zustände korrekt gelesen werden können.

Spaced-Repetition-Logik

Die Spaced-Repetition-Logik ist vollständig in LearningContext.tsx implementiert. Sie basiert auf einem 6-Fach-Lernkartei-System: Karten wandern bei richtiger Antwort in ein höheres Fach (längeres Wiederholungsintervall), bei falscher Antwort immer zurück in Fach 1.

Das 6-Fach-System und seine Intervalle

Die Intervalle sind in constants/theme.ts als einziger zentraler Wert definiert:

export const BOX_INTERVALS = [0, 1, 3, 7, 14, Infinity];
//  Index:                     0  1  2   3   4       5
//  Fach:                      1  2  3   4   5       6

Der Wert an Position newBox - 1 gibt an, wie viele Tage nach dem heutigen Datum die Karte als nächstes fällig ist. Fach 6 mit Infinity bedeutet: nie wieder fällig – die Karte ist vollständig gelernt.

* Fach 1 hat rechnerisch Intervall 0. In der Praxis wird bei falscher Antwort immer morgen gesetzt, um eine Sofortwiederholung am selben Tag zu verhindern.

Falsche Antwort = immer zurück auf Fach 1 Egal aus welchem Fach eine Karte kommt: bei falscher Antwort landet sie in Fach 1 mit nextDate = morgen. Die Karte erscheint also frühestens am nächsten Tag wieder – nie sofort erneut in derselben Sitzung.

Prozessablauf: markCard(vocabId, lang, correct)

Die Kernfunktion wird nach jeder Kartenbewertung aufgerufen. Sie läuft komplett innerhalb eines setState(s => …)-Callbacks ab:

1
Aktuellen Fortschritt laden
current = s.progress["{vocabId}_{lang}"] ?? defaultProgress()
Neu-Karten haben keinen Eintrag; defaultProgress() liefert {box:1, lastReviewed:null, nextDate:null, correctCount:0, incorrectCount:0}.
2
Neues Fach berechnen
Richtig: newBox = Math.min(current.box + 1, 6) – ein Fach höher, maximal Fach 6.
Falsch: newBox = 1 – immer zurück in Fach 1.
3
nextDate berechnen
Falsch: nextDate = morgen (Fach 1, aber nicht sofort).
Richtig & newBox = 6: nextDate = null (gelernt, nie wieder fällig).
Richtig & newBox < 6: nextDate = heute + BOX_INTERVALS[newBox - 1] Tage
Beispiel: Karte in Fach 3 richtig → newBox=4 → nextDate = heute + 7 Tage.
4
Tagesstatistik aktualisieren (dailyStats)
dailyStats[lang] speichert Datum und Zähler. Wenn das gespeicherte Datum gleich heute ist, wird count + 1 gesetzt; sonst wird der Zähler auf 1 zurückgesetzt (neuer Tag). Diese Statistik begrenzt, wie viele fällige Karten heute noch eingeblendet werden.
5
Neue-Karten-Statistik aktualisieren (dailyNewStats)
Wenn current.lastReviewed === null (erste Bewertung dieser Karte), wird auch dailyNewStats[lang].count + 1 gesetzt. Dieser Zähler begrenzt, wie viele neue Karten heute noch eingeführt werden.
6
State zurückgeben
Neuer Progress-Eintrag: {box: newBox, lastReviewed: jetzt (ISO), nextDate, correctCount ± 1, incorrectCount ± 1}. Alles per Spread-Syntax als neues Objekt – der State wird nie mutiert.

Prozessablauf: isCardDue(progress)

Hilfsfunktion (nicht direkt im Context-API, intern genutzt). Bestimmt, ob eine Karte heute fällig ist:

→
Fach 6 (box >= 6): Sofort false. Gelernte Karten erscheinen nie wieder.
→
Neue Karte (lastReviewed === null): Sofort false. Noch nie gesehene Karten laufen über getNewCards(), nicht über getDueCards().
→
Normalpfad (nextDate vorhanden): return localDateStr(today) >= progress.nextDate. Einfacher String-Vergleich im Format YYYY-MM-DD – da das Format lexikografisch sortierbar ist, reicht ein String-Vergleich.
!
Legacy-Karten (kein nextDate): Rückfall auf intervalbasierte Berechnung: diffDays = (now - lastReviewed) / msPerDay. Ist diffDays >= BOX_INTERVALS[box - 1], gilt die Karte als fällig.

Prozessablauf: getDueCards(lang)

Wird von Home.tsx (Tagesübersicht) und Learn.tsx (Sitzungsstart) aufgerufen:

1
Tages-Restlimit berechnen
alreadyDone = dailyStats[lang].date === heute ? count : 0
remaining = dailyCardLimit - alreadyDone
Ist dailyCardLimit = 0 → keine Beschränkung (Infinity).
2
Alle fälligen Karten filtern
Über alle Vokabeln der Sprache (eingebaut + custom) wird isCardDue(progress) geprüft.
3
Sortierung: Box aufsteigend, dann nextDate aufsteigend
Karten in Fach 1 (Wiederholungen falscher Antworten) erscheinen zuerst. Bei gleichem Fach kommt die älteste Fälligkeit zuerst.
4
Limit anwenden: cards.slice(0, remaining)
Die Rückgabe enthält maximal so viele Karten, wie das Tageslimit noch zulässt.

Prozessablauf: getNewCards(lang)

1
Limit 0? Sofort leeres Array zurückgeben (Benutzer möchte keine neuen Karten).
2
Bereits heute eingeführt (dailyNewStats)
introducedToday = dailyNewStats[lang].date === heute ? count : 0
remaining = dailyNewCardLimit - introducedToday. Bei Limit -1 → Infinity (unbegrenzt).
3
Neue Karten filtern
Alle Vokabeln der Sprache, bei denen getCardProgress().lastReviewed === null. Die Reihenfolge entspricht der Reihenfolge im Vokabelarray (eingebaut zuerst, dann custom).
4
cards.slice(0, remaining) – maximal so viele neue Karten wie das Tageslimit zulässt.

Tagesgrenzen und Interaktion zwischen den Limits

Es gibt zwei unabhängige Tageszähler, die beide gleichzeitig wirken:

  • dailyCardLimit / dailyStats: Wie viele fällige Karten können heute noch gezeigt werden? Jede Bewertung (richtig oder falsch) zählt hier rein.
  • dailyNewCardLimit / dailyNewStats: Wie viele neue Karten (erstmalig gesehen) können heute noch eingeführt werden? Zählt nur bei der allerersten Bewertung einer Karte.

Beide Zähler werden täglich automatisch zurückgesetzt: beim ersten Zugriff nach Tageswechsel erkennt der Code, dass das gespeicherte Datum nicht mehr dem heutigen entspricht, und startet bei 0.

Learn.tsx – Sitzungs-Zustandsmaschine

Die komplexeste Seite der App. Sie verwaltet einen lokalen Sitzungs-State und durchläuft dabei klar definierte Phasen:

Zeigt die vier Modus-Buttons: Fällige Karten, Alle Karten, Quiz, Tippen. Jeder Button kennt seine Kartenquelle (getDueCards vs. getNewCards + getDueCards vs. alle).
②
startSession(mode) – Übergang Lobby → aktive Sitzung.
Lädt die Kartenliste je nach Modus. Bei Fällige Karten: getDueCards() + getNewCards(), gemischt. Bei Alle Karten: alle Vokabeln der Sprache. Bei Quiz und Tippen: fällige + neue Karten. Setzt sessionCards, currentIndex = 0.
③
Aktive Sitzung – Kartendurchlauf
Drei Render-Zweige je nach Modus: Karteikarten-Flip (FlashCard.tsx), Multiple-Choice-Quiz (4 Optionen, 1 richtige), Tippen (Freitexteingabe + optionale Tipptoleranz via Levenshtein).
Tastatur-Shortcuts werden über einen globalen keydown-Listener implementiert (Enter = Karte umdrehen / bestätigen, Pfeiltasten = richtig/falsch). Der Listener wird beim Sitzungsende entfernt.
④
Karte bewerten: markCard()
Nach jeder Bewertung wird markCard(id, lang, correct) im Context aufgerufen. Der nächste currentIndex++-Schritt blendet die nächste Karte ein.
⑤
Sitzung beendet – Auswertungsansicht
Wenn currentIndex >= sessionCards.length, wechselt die Darstellung zur Ergebnisansicht (richtige, falsche, Gesamtzahl). Bei Fällig-Karten-Sitzungen wird recordTrainingDay(lang) aufgerufen (idempotent – nur einmal pro Tag).

Die Tipptoleranz im Eingabe-Modus nutzt einen Levenshtein-Algorithmus, der direkt in Learn.tsx implementiert ist. Eine Antwort wird als „toleriert richtig" gewertet, wenn die Editierdistanz ≤ 2 ist und die Eingabe mindestens 4 Zeichen lang ist.

Sprachausgabe (Web Speech API)

Die Funktion speak(text, lang) nutzt die Web Speech API (SpeechSynthesisUtterance). Ab v1.2.1 wird die Systemstimme explizit per pickVoice(lang) ausgewählt, statt dem Browser die Wahl zu überlassen:

// Modulebene – einmalig beim Laden, danach per Event aktualisiert
let _voices: SpeechSynthesisVoice[] = [];
speechSynthesis.addEventListener('voiceschanged', () => {
  _voices = speechSynthesis.getVoices();
});

// Sucht zuerst exakte Locale (en-US), dann Sprachpräfix (en)
function pickVoice(lang: string): SpeechSynthesisVoice | null {
  const voices = _voices.length ? _voices : speechSynthesis.getVoices();
  return voices.find(v => v.lang === lang)
    ?? voices.find(v => v.lang.startsWith(lang.split('-')[0]))
    ?? null;
}

Der Voice-Cache auf Modulebene umgeht einen bekannten Browser-Bug (vor allem Chrome), bei dem getVoices() beim ersten synchronen Aufruf noch eine leere Liste zurückgibt. Das voiceschanged-Event signalisiert, wann die Stimmen tatsächlich verfügbar sind.

Filterung per useMemo auf Basis von Suchtext, Kategorie und Fach – die gefilterte Liste wird nur neu berechnet, wenn sich die Filterwerte oder die Vokabelliste ändert. PWA & Service Worker

Workbox-Strategie (vite.config.ts)

Asset-TypStrategieCache-Dauer
JS, CSS, Fonts, Bilder (Precache)CacheFirst1 Jahr
index.html (SPA-Fallback)Precache (kein Ablauf)dauerhaft bis Update

clientsClaim: true und skipWaiting: true stellen sicher, dass ein neuer Service Worker sofort alle offenen Tabs übernimmt. Das ist auf iOS besonders wichtig, wo Tabs den alten Worker sonst sehr lange behalten können.

Service-Worker-Lebenszyklus und Update-Ablauf

1
Registrierung beim App-Start
navigator.serviceWorker.register('/sw.js'). Im Dev-Modus (Vite HMR) wird kein SW registriert.
2
Stündlicher Update-Check
setInterval() löst alle 60 Minuten registration.update() aus – aber nur wenn navigator.onLine === true, um Fehlermeldungen im Offline-Betrieb zu vermeiden.
3
Vordergrund-Event: visibilitychange
Wenn die App aus dem Hintergrund zurückkommt (Tab wieder sichtbar) und online ist, wird sofort ein Update-Check ausgelöst. Falls kein SW mehr registriert ist (iOS hat ihn beendet), wird einmalig ein Reload versucht.
4
iOS bfcache: pageshow
Safari kann Seiten im Back-/Forward-Cache einfrieren. Der pageshow-Listener mit event.persisted-Check erkennt diesen Fall und löst ebenfalls einen Update-Check aus.
5
Neuer SW verfügbar
Da skipWaiting: true gesetzt ist, aktiviert sich ein neuer SW sofort. Mit clientsClaim: true übernimmt er alle offenen Fenster. Beim nächsten Seitenaufruf ist die neue Version aktiv.

swOverlay.ts

Hilfsfunktion für die Diagnose. Blendet ein temporäres, terminal-ähnliches Overlay ein (grüner Text auf schwarzem Hintergrund), das den aktuellen SW-Zustand, Online-Status und Cache-Belegung anzeigt. Verschwindet nach 8 Sekunden oder bei Antippen. Nur über das SWStatusPanel in den Einstellungen erreichbar.

App erweitern

Neue Vokabeln zu einer bestehenden Sprache hinzufügen

Die Datei src/data/vocabulary_en.ts (bzw. vocabulary_es.ts) erweitern:

// Eintrag zum VOCABULARY_EN Array hinzufügen:
{
  id: 'eindeutige_id',      // Muss eindeutig sein – z.B. 'tree', 'to_swim_en'
  german: 'der Baum',
  translation: 'tree',
  emoji: '🌳',
  category: 'Natur & Wetter',
  inflections: '[trees]',   // optional
}

Neue Kategorie hinzufügen

  1. Den neuen Kategorienamen als Union-Member in src/data/vocabulary_types.ts zum Category-Typ hinzufügen.
  2. Den gleichen String in das ALL_CATEGORIES-Array in src/data/vocabulary.ts eintragen.

Neue Sprache hinzufügen

Die App unterstützt beliebig viele Sprachen. Eine Schritt-für-Schritt-Anleitung dazu ist in docs/neue-sprache-hinzufuegen.md beschrieben.

Zusammengefasst sind folgende Schritte nötig:

  1. src/data/vocabulary_XX.ts anlegen und mit VOCABULARY_XX befüllen.
  2. Re-Export in src/data/vocabulary.ts ergänzen.
  3. Neuen Spracheintrag in src/constants/languages.ts zu LANGUAGE_CONFIG hinzufügen (Label, Flag, Abbr, speechCode, placeholder).
  4. Language-Typ in languages.ts um den neuen Wert erweitern.
  5. Im LearningContext: Initialen State um die neue Sprache ergänzen (customVocabularyXX, dailyStats, dailyNewStats, trainingLog).
Kein zentrales Routing-Update nötig Seiten wie Home, Statistics und Vocabulary lesen Sprachen dynamisch aus ALL_LANGUAGES (aus constants/languages.ts). Nach dem Hinzufügen einer Sprache erscheint sie automatisch in der Sprachauswahl und in den Filtern, ohne dass Seiten-Code angepasst werden muss.

Neuen Lernmodus hinzufügen

Alle Lernmodi sind in src/pages/Learn.tsx in einem einzigen SessionMode-Union-Typ definiert. Ein neuer Modus erfordert:

  1. SessionMode-Union um den neuen Wert erweitern.
  2. Button im Lobby-Screen ergänzen (Array mit mode, colors, emoji, title, sub).
  3. startSession: Kartenauswahl-Logik für den neuen Modus implementieren.
  4. Render-Zweig für den neuen Modus vor dem Karteikarten-Fallback hinzufügen.
